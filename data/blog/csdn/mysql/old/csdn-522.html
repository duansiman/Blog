\n
\n

      触发器是一种特殊类型的存储过程，它不同于之前的我们介绍的存储过程。触发器主要是通过事件进行触发被自动调用执行的。而存储过程可以通过存储过程的名称被调用。
\n       触发器的主要作用就是能够实现比参照完整性更为复杂的两张表或多张表之间的数据的完整性和一致性，从而保证表中的数据的变化符合数据库设计者确定的业务规则。
\n       触发器对表进行插入、更新、删除的时候会自动执行的特殊存储过程。触发器一般用在check约束更加复杂的约束上面。
\n       触发器和普通的存储过程的区别是：触发器是当对某一个表进行操作。诸如：update、insert、delete这些操作的时候，系统会自动调用执行该表上对应的触发器。
\n       SQL Server 2005中触发器可以分为两类：DML触发器和DDL触发器，其中DDL触发器它们会影响多种数据定义语言语句而激发，这些语句有create、alter、drop语句。

\n
\n\n

DML触发器分为：

\n\n
\n

1、 after触发器（之后触发）
\n       a、 insert触发器
\n       b、 update触发器
\n       c、 delete触发器
\n 2、 instead of 触发器 （之前触发）
\n 3、区别
\n       after触发器：要求只有执行某一操作insert、update、delete之后触发器才被触发，且只能定义在表上。
\n       instead of触发器：表示并不执行其定义的操作（insert、update、delete）而仅是执行触发器本身。既可以在表上定义instead of触发器，也可以在视图上定义。

\n
\n\n

inserted 表和 deleted 表

\n\n
\n

      在使用触发器时，SQL Server会为每个触发器建立两个特殊的临时表，即inserted表和deleted表。这两个表存储在内存中，与创建触发器的表具有相同的结构，由系统维护和管理，不允许用户对其进行修改。每个触发器只能访问自己的临时表，触发器执行完毕以后，两表会自动释放。
\n       （1）inserted表用于存储insert或update语句所影响的行的副本。当执行insert或update操作时，新的数据行同时被添加到激活触发器的基本表和inserted表中。
\n       （2）deleted表用于存储delete或update语句所影响的行的副本。当执行delete或update操作时，指定的原数据行被从基本表中删除，然后被转移到deleted表中。一般来说，在基本表和deleted表中不会存在相同的数据行。
\n        说明：
\n       update操作分为两步：首先将基本表中修改的原数据行转移到deleted表中，然后从inserted表中复制修改后的新数据行到基本表中。也就是说，对于update操作，deleted表中存放的是修改之前的旧值，inserted表中存放的是修改之后的新值。

\n
\n\n
\n\n

触发器创建语法四要素：

\n\n
\n

1.监视地点(table)
\n 2.监视事件(insert/update/delete)
\n 3.触发时间(after/before)
\n 4.触发事件(insert/update/delete)

\n
\n\n

\n\n
delimiter &&\ncreate trigger triggerName\nafter/before insert/update/delete on 表名\nfor each row #这句话在mysql是固定的\nbegin\nsql语句;\nend&&

如果我们在没使用触发器之前：假设我们现在卖了3个商品1，我们需要做两件事

\n\n
\n

1.往订单表插入一条记录

\n \n

insert into o(gid,much) values(1,3);

\n \n

2.更新商品表商品1的剩余数量

\n \n

update g set num=num-3 where id=1;

\n
\n\n

创建触发器：

\n\n\n\n
delimiter &\nmysql> create trigger trg1\n -> after insert on o\n -> for each row\n -> begin\n -> update g set num = num -3 where id = 1;\n -> end&
\n\n

执行：

\n\n\n\n
insert into o(gid,much) values(1,3)$
\n\n

结果：

\n\n

会发现商品1的数量变为7了，说明在我们插入一条订单的时候，触发器自动帮我们做了更新操作。

\n\n

但现在会有一个问题，因为我们触发器里面num和id都是写死的，所以不管我们买哪个商品，最终更新的都是商品1的数量。比如：我们往订单表再插入一条记录：insert into o(gid,much) values(2,3),执行完后会发现商品1的数量变4了，而商品2的数量没变，这样显然不是我们想要的结果。我们需要改改我们之前创建的触发器。

\n\n

我们如何在触发器引用行的值，也就是说我们要得到我们新插入的订单记录中的gid或much的值。

\n\n

对于insert而言，新插入的行用new来表示，行中的每一列的值用new.列名来表示。

\n\n

所以现在我们可以这样来改我们的触发器

\n\n

create trigger tg2
\nafter insert on o
\nfor each row
\nbegin
\nupdate g set num=num-new.much where id=new.gid;(注意此处和第一个触发器的不同)
\nend$

\n\n

第二个触发器创建完毕，我们先把第一个触发器删掉

\n\n

drop trigger tg1$

\n\n

再来测试一下，插入一条订单记录：insert into o(gid,much) values(2,3)$

\n\n

执行完发现商品2的数量变为7了，现在就对了。

\n\n

现在还存在两种情况：

\n\n

1.当用户撤销一个订单的时候，我们这边直接删除一个订单，我们是不是需要把对应的商品数量再加回去呢？

\n\n

2.当用户修改一个订单的数量时，我们触发器修改怎么写?

\n\n

我们先分析一下第一种情况：

\n\n

监视地点：o表

\n\n

监视事件：delete

\n\n

触发时间：after

\n\n

触发事件：update

\n\n

对于delete而言：原本有一行,后来被删除，想引用被删除的这一行，用old来表示，old.列名可以引用被删除的行的值。

\n\n

那我们的触发器就该这样写：

\n\n

create trigger tg3

\n\n

after delete on o

\n\n

for each row

\n\n

begin

\n\n

update g set num = num + old.much where id = old.gid;(注意这边的变化)

\n\n

end$

\n\n

创建完毕。

\n\n

再执行delete from o where oid = 2$

\n\n

会发现商品2的数量又变为10了。

\n\n

第二种情况：

\n\n

监视地点：o表

\n\n

监视事件：update

\n\n

触发时间：after

\n\n

触发事件：update

\n\n

对于update而言：被修改的行，修改前的数据，用old来表示，old.列名引用被修改之前行中的值；

\n\n

修改的后的数据，用new来表示，new.列名引用被修改之后行中的值。

\n\n

那我们的触发器就该这样写：

\n\n

create trigger tg4

\n\n

after update on o

\n\n

for each row

\n\n

begin

\n\n

update g set num = num+old.much-new.much where id = old/new.gid;

\n\n

end$

\n\n

先把旧的数量恢复再减去新的数量就是修改后的数量了。

\n\n

我们来测试下：先把商品表和订单表的数据都清掉，易于测试。

\n\n

假设我们往商品表插入三个商品，数量都是10，

\n\n

买3个商品1：insert into o(gid,much) values(1,3)$

\n\n

这时候商品1的数量变为7；

\n\n

我们再修改插入的订单记录: update o set much = 5 where oid = 1$

\n\n

我们变为买5个商品1，这时候再查询商品表就会发现商品1的数量只剩5了，说明我们的触发器发挥作用了。
